# Build a VSCP TCP server on STM32F103 with Wiznet IP20

This is a simple implementation of a VSCP TCP server on an STM32F103 microcontroller using the Wiznet IP20 Ethernet module. The server listens for incoming TCP connections and responds with a welcome message and after that it handles VSCP events.

1.) Many items of the implementation are dependent on a reliable one millisecond tick. This is used for timeouts and other timing related functions. The implementation uses the HAL library for the STM32F103, which provides a SysTick timer that can be configured to generate an interrupt every 1 ms. The SysTick_Handler function is implemented to increment a global tick counter, which can be used for timing. Read this clock with a call to HAL_GetTick()

2.) Some sort of persistent storage is needed for VSCP standard registers and for use3r registers that need to be retained across power cycles. The STM32F103 has 32 KB of internal flash memory, which can be used for this purpose. A simple flash storage driver is implemented to read and write data to the flash memory. The last 2 KB of the flash memory is reserved for storage, while the first 30 KB is used for the firmware. The driver provides functions to erase, write, and read data from the flash storage, and it handles the necessary constraints of flash memory (such as erasing before writing and 16-bit alignment). This allows for reliable storage of configuration data and other information that needs to persist across power cycles.

3.) The Wiznet IP20 module is configured using two letter commands sent over UART. The module is set to operate in TCP server mode, and the necessary network parameters (IP address, subnet mask, gateway) are configured. The implementation includes functions to send commands and receive responses from the module. Communication is set up to be blocking, but with a timeout to prevent hanging if the module does not respond. And change to a non blocking interrupt driven receive after the initial configuration. Commands are consumed from a fifo buffer filled by the UART receive interrupt handler.

4.) The main loop of the server checks for incoming TCP connections and handles them accordingly. When a client connects, a welcome message is sent, and the server enters a loop to handle incoming data from the client. The implementation includes a simple protocol for handling VSCP events, where the client can send commands to trigger specific actions on the server. The server also handles disconnections and errors gracefully, ensuring that resources are properly released.

Two special strings are used in the implementation: <CONNECT> and <DISCONNECT>. These are printed to the console when a client connects or disconnects, respectively. This allows for easy monitoring of client connections and disconnections.

5.) After a module has connected the VSCP tcp/ip link protocol is used. This protocol is implemented in the vscp-link-protocol.h/vscp-link-protocol.c files of that is available in the [vscp-firmware repository](https://github.com/grodansparadis/vscp-firmware). This repository is added to the project as a git submodule. The protocol implementation is used to parse incoming commands and VSCP events and to format responses and outgoing events. The server can handle basic VSCP commands such as sending events, receiving events, and querying the server status.

6.) The actual implemention of the VSCP link protocol is done suing callbacks functions. The main loop of the server checks for incoming data and when a complete command is received, it calls the appropriate callback function to handle the command. This allows for a clean separation of the protocol handling logic from the main server loop, making the code easier to maintain and extend in the future.

7.) When the VSCP tcp/ip link protocol is in place the functionality of the node needs to be implemented. The VSCP blink demo is a standard demo that works on several platforms and have functionality around a single LED that can be turned on and off and blink. The implementation of the VSCP link protocol is used to handle incoming commands to control the LED, as well as to send events when the LED state changes. This allows for a simple demonstration of the capabilities of the VSCP TCP server and provides a basis for further development and customization. 

If you only want to implement a protocol converter that converts between a serial protocol and the VSCP link protocol you are done here. All you need to do is to send incoming events on the format and also return events in the format of the VSCP link protocol. The client can then be a standard VSCP client that connects to the TCP server and sends and receives VSCP events.

If the node needs other functionality it is now time to implement this functionality. In this case read/write registers of the node, handle specific commands, implement a decision matrix, etc. 

8.) Implement any additional functionality required for the node. This could include reading and writing registers, handling specific commands, implementing a decision matrix, or other custom features. The vscp-firmware-level2.h/vscp-firmware-level2.c files provide a framework for implementing additional functionality, and the callback functions can be used to integrate this functionality into the main server loop. This allows for a flexible and extensible implementation that can be tailored to the specific needs of the application.
